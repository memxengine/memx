# F111.2 — Admin API-key panel + multi-origin CORS for integrations

> Lukker hullet mellem F111.1 (DB-backed `trail_<64hex>` Bearer keys, allerede shippet) og en tredjepart-integration der kan kalde Trail's HTTP API fra et andet origin. Tre dele: (1) admin-panel under Settings → Account hvor curator kan generere/liste/tilbagekalde API-keys, (2) `TRAIL_ALLOWED_ORIGINS` env-variabel der lader os whitelist'e fx Sanne Andersen's site på localhost:3001 uden at hard-code originet, (3) `docs/INTEGRATION-API.md` der dokumenterer endpoints + auth + curl-eksempler så cc-sessioner i andre repos kan integrere uden at læse Trail's source. Tier: alle tenants. Effort: Small — 0.5–1 dag. Status: Planned.

## Problem

Sanne Andersen's website er Trail's første eksterne integrations-customer: en AI-assistent på hendes site der trækker svar fra hendes Trail KB i realtid og booker tider for kunder. Sanne-andersen-cc skal lave selve chat-fronten på localhost mens vi udvikler. Det udstiller tre konkrete huller mellem hvad F111.1 leverer og hvad en integrator faktisk kan bruge:

1. **Admin-UI mangler.** F111.1 gav os `GET/POST/DELETE /api/v1/api-keys` men ingen UI. For at oprette en key skal man enten åbne devtools mod et levende admin-session eller `curl -b session=<cookie>` det. Begge er friction-tunge nok til at vi mocker chat-integrationen i stedet for at bygge den ægte vej.
2. **CORS-whitelist er hard-coded til admin-origin.** `apps/server/src/app.ts` line 56-74 tillader kun `APP_URL` (default `http://localhost:3030`) + browser-extension-prefixer. Sanne-andersen på localhost:3001 ville få `Access-Control-Allow-Origin`-mismatch ved første preflight, og vi har ingen env-knap til at tilføje origins uden at redigere koden.
3. **Ingen integration-doc.** En cc-session i et andet repo har ingen kanonisk reference for "hvad er endpoint URL'en, hvad er header-formatet, hvad er fejlkoderne, hvad er body-shapen". Den vil ende med at læse `apps/server/src/routes/chat.ts` direkte og gætte. Det er ineffektivt og fejlbarligt — body-shapen ændrer sig over tid, og external integrators skal kun se den **stabile** kontrakt.

Uden disse tre dele bliver Sanne-integrationen mocket frem til prod, vi mister learning-loopet på den ægte API, og vi forsinker det øjeblik hvor en tredjepart-cc kan bygge mod Trail uden Christian-i-loop'en.

## Secondary Pain Points

- **Last-used-at synlighed.** F111.1 stamper `apiKeys.last_used_at` best-effort ved hver Bearer-request, men der er intet UI der viser det. En curator kan ikke se "blev denne key brugt i sidste måned?" og dermed heller ikke beslutte sig for at revoke ubrugte keys.
- **Key-naming-disciplin.** Uden et UI er der intet der tvinger curator til at give keys navne der siger noget. CLI-folk laver let "test", "sanne", "x" som navne — UI'et kan tilskynde noget mere informativt ved at vise eksempler ("e.g. sanne-andersen.dk-prod") og default-foreslå dato + integration-type.
- **Plaintext-key håndtering.** F111.1's POST returnerer raw `trail_<64hex>` ÉN gang. Hvis curator lukker browseren / kopierer forkert, må de revoke + generere ny. UI bør håndhæve at modal'en ikke kan lukkes uden at curator har trykket "Copy" eller eksplicit "Jeg har gemt den".
- **Origin-validering.** En CSV-baseret env er fleksibel men også easy-to-misconfigure. Vi bør validere at hver entry er en velformet origin (`scheme://host[:port]`) ved boot og logge advarsel ved ugyldige rækker, ikke fejle silently.

## Solution

Tre delkomponenter, alle små:

### 1. Admin UI — section under Settings → Account

Filer:
- `apps/admin/src/api.ts` — wrappere for `listApiKeys`, `createApiKey({ name })`, `revokeApiKey(id)`, samt en `ApiKeyRow` type.
- `apps/admin/src/panels/settings-account.tsx` — ny `<section>` med listen + actions.
- `apps/admin/src/components/modal.tsx` — genbrug eksisterende `Modal` + `ModalButton` til create-flow + revoke-bekræftelse. **Aldrig** `window.confirm` / `window.prompt` (CLAUDE.md hard rule).

UX-flow:
1. Section overskrift: "API Keys" + sublabel: "Brug Bearer tokens til at integrere Trail i eksterne sites og scripts."
2. Hvis ingen keys: tom-state med "Du har endnu ikke oprettet en API key" + primær knap.
3. Hvis keys eksisterer: tabel m/ kolonner Name, Last used, Created, Actions ([Revoke]).
4. "Create new key" knap → modal: input til Name (required, validate non-empty), foreslået placeholder med eksempler. Submit → POST → modal viser raw key i `<code>`-blok m/ Copy-knap, advarsel "Du ser denne key kun ÉN gang. Gem den nu i din password manager eller integration's config." Modal kan kun lukkes via eksplicit "Jeg har gemt den"-knap (ikke X eller esc — same anti-loss-pattern som vi bruger i Save-as-Neuron).
5. Revoke → bekræftelses-modal "Tilbagekald X? Eksisterende integrationer der bruger denne key vil få 401." → POST DELETE → row fjernes.

Section sidder under Profile, Tenant og Ambient sections i settings-account.tsx — samme `<section>` + `<h2>` mønster, ingen ny route, ingen nav-ændring. Det er den mindste flade for redesignet at re-arrangere senere.

### 2. CORS multi-origin via env

Fil: `apps/server/src/app.ts`.

Tilføj en helper øverst der parser `TRAIL_ALLOWED_ORIGINS` (CSV) ved boot:

```typescript
function loadConfiguredOrigins(): string[] {
  const raw = process.env.TRAIL_ALLOWED_ORIGINS;
  if (!raw) return [];
  return raw
    .split(',')
    .map((s) => s.trim())
    .filter((s) => {
      if (!s) return false;
      try {
        const u = new URL(s);
        // Origin = scheme://host[:port], no path/query/hash. Reject anything
        // else so we fail loud at boot rather than silently allowing
        // misconfigured entries through.
        if (u.pathname !== '/' && u.pathname !== '') return false;
        return true;
      } catch {
        console.warn(`[cors] ignoring invalid origin in TRAIL_ALLOWED_ORIGINS: ${s}`);
        return false;
      }
    })
    // Strip trailing slash so '/' vs '' doesn't break exact match.
    .map((s) => s.replace(/\/$/, ''));
}
const CONFIGURED_EXTRA_ORIGINS = loadConfiguredOrigins();
```

Eksisterende `cors({ origin: ... })`-callback udvides:

```typescript
const allowed = [
  process.env.APP_URL ?? 'http://localhost:3030',
  'http://localhost:3030',
  'http://127.0.0.1:3030',
  ...CONFIGURED_EXTRA_ORIGINS,
];
if (allowed.includes(origin)) return origin;
// (extension prefixes uændret)
```

Default-adfærd er uændret. Sanne sætter `TRAIL_ALLOWED_ORIGINS=http://localhost:3001` i sin lokale `.env` og er på.

### 3. `docs/INTEGRATION-API.md`

Sider for cc-sessioner i andre repos der bygger mod Trail's HTTP API:

- **Auth** — header-format, hvor man får en key (link til admin-UI), what scope keys have (per-user, kan revokes), 401-respons-shape.
- **Endpoints** — kun de stabile/integrations-relevante: `POST /api/v1/chat`, `GET /api/v1/knowledge-bases`, `GET /api/v1/api-keys` (selvbetjening), framework for fremtidige endpoints. **Ikke** intern admin-API (fx queue-routes) — de er ikke stabile kontrakter.
- **Body-shape pr. endpoint** — request + response, error-koder + payload-shape (inkl. F156 Phase 1's `code: session_turn_cap_reached`).
- **Conversation-flow** — sessionId-pinning, multi-turn memory, turn-cap UX (`turnsUsed` / `turnsLimit` på response).
- **Rate-limits** — vi har ingen pr. key-limit i v1; Phase 2's F44 Usage Metering vil indføre. Note "fair use" + at credits-systemet (F156) er den primære økonomiske control.
- **Quickstart** — env-vars + en kort fetch-snippet til en chat-roundtrip.

Ligger i `docs/` (ikke i feature-folder) fordi den er en kontrakts-doc, ikke et plan-doc.

## Non-Goals

- **Tenant-scoped keys i v1.** Keys er per-user (skabt af en logget-ind user, scope ranger på den user's tenant). En "service-account user" pattern dækker integrationer; vi indfører ikke tenant-level keys uden user før F40.2 multi-tenancy faktisk er live og forskellen mellem user/tenant-scope giver praktisk forskel.
- **Per-key scope-restriction.** Keys arver hele user'ens autorisation. Vi tilføjer ikke "denne key må kun read /chat" i v1; det er Phase 2-territorium (sammen med F44).
- **Per-key rate-limit i admin.** F44 Usage Metering håndterer det; lige nu er den primære økonomiske control credits (F156) der dræner ved misbrug.
- **OAuth / PKCE.** Bearer tokens er enkle, audit-bare, revoke-bare og dækker integrations-use-casen. OAuth-bro mod Trail er Phase 3-territorium hvis nogensinde.
- **Key-rotation cron.** Keys udløber ikke automatisk i v1. Curator revoker manuelt når de vil.
- **CORS wildcard / `*`.** Vi tillader **aldrig** `*` i `Access-Control-Allow-Origin` på et bearer-credentialed endpoint — det er en sikkerhedsanti-pattern. Env'en accepterer kun konkrete origins.
- **Origin-bound keys.** En key fungerer fra et hvilket som helst origin der har CORS-clearance — vi binder ikke en key til ét specifikt origin i v1. Fremtidigt mulighed når vi har flere reelle integrationer at lære af.

## Technical Design

### Schema — uændret

F111.1's `apiKeys`-tabel dækker hele behovet (id, tenantId, userId, name, keyHash, lastUsedAt, createdAt, revokedAt). Ingen migration.

### Server — kun CORS-helper

`apps/server/src/app.ts`:
- Ny `loadConfiguredOrigins()` helper øverst i `createApp` (eller modul-niveau, computed once).
- `allowed`-array i `cors({ origin: ... })` udvides med de parsed origins.
- Intet andet i routes — F111.1's auth-middleware + api-keys-routes er allerede i orden.

### Admin UI — settings-account section

Snippet (kondenseret — fuld impl i commit):

```tsx
function ApiKeysSection() {
  const [keys, setKeys] = useState<ApiKeyRow[] | null>(null);
  const [createOpen, setCreateOpen] = useState(false);
  const [newKey, setNewKey] = useState<{ key: string; name: string } | null>(null);
  const [revokeTarget, setRevokeTarget] = useState<ApiKeyRow | null>(null);

  useEffect(() => { listApiKeys().then(setKeys); }, []);

  return (
    <section>
      <h2>API Keys</h2>
      <p class="text-sm text-fg-muted mb-3">
        Brug Bearer tokens til at integrere Trail i eksterne sites og scripts.
      </p>
      {keys === null ? <CenteredLoader /> :
       keys.length === 0 ? <EmptyState onCreate={() => setCreateOpen(true)} /> :
       <ApiKeyTable rows={keys} onRevoke={setRevokeTarget} />}
      <button onClick={() => setCreateOpen(true)}>Generér ny API key</button>

      <CreateKeyModal
        open={createOpen}
        onClose={() => setCreateOpen(false)}
        onCreated={(k) => { setNewKey(k); setCreateOpen(false); refresh(); }}
      />
      <RevealKeyModal
        keyData={newKey}
        onAcknowledge={() => setNewKey(null)}
      />
      <RevokeConfirmModal
        target={revokeTarget}
        onClose={() => setRevokeTarget(null)}
        onRevoked={() => { setRevokeTarget(null); refresh(); }}
      />
    </section>
  );
}
```

`RevealKeyModal` har **kun** "Jeg har gemt den"-knap (ingen X, ingen esc-handler) — samme anti-loss pattern som CLAUDE.md's "knapper SKAL give feedback"-regel.

### `docs/INTEGRATION-API.md`

Skitse:

```markdown
# Trail HTTP API for Integrators

## Authentication
Header: `Authorization: Bearer trail_<64hex>`

## Endpoints
### POST /api/v1/chat
Body: { message, knowledgeBaseId, sessionId? }
Returns: { answer, renderedAnswer, citations[], sessionId, backend, model, turnsUsed, turnsLimit }
Errors: 401 (invalid key), 429 ({ code: 'session_turn_cap_reached', ... }), 500 (LLM)

### GET /api/v1/knowledge-bases
Returns: KnowledgeBase[]

### GET /api/v1/api-keys
Returns: ApiKeyRow[] (without raw key)

## Conversation lifecycle
- First call: omit sessionId → server creates one + returns it.
- Subsequent calls: send returned sessionId.
- Hit turnsUsed === turnsLimit → start new chat (omit sessionId).

## Quickstart
```bash
TRAIL_API_BASE=http://localhost:3000
TRAIL_API_KEY=trail_xxx
TRAIL_KB_ID=sanne-andersen
```
```ts
const res = await fetch(`${TRAIL_API_BASE}/api/v1/chat`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${TRAIL_API_KEY}`,
  },
  body: JSON.stringify({ message, knowledgeBaseId: TRAIL_KB_ID, sessionId }),
});
const data = await res.json();
// data.answer, data.sessionId — pin for next turn
```

## Rate limits
Ingen pr-key-limit i v1. Credits (F156) er primær økonomisk control.
```
